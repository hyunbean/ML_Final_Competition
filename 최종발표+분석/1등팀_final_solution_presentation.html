04Feature Engineering

영수증을 여러 관점의
고객 프로필로 바꿨다

핵심은 상품명이나 브랜드명을 그대로 모델에 던지는 것이 아니라, 고객의 구매 습관과 구매 맥락을 모델이 이해할 수 있는 숫자 source로 바꾸는 것이었다.

고객 행동 요약

Basic Aggregation

구매 횟수, 총 구매액, 평균 구매액, 환불 비율, 주말/평일/시간대 비율처럼 고객의 전체 소비 습관을 요약했다. 거래 100만 줄을 고객 3만 명의 프로필로 바꾸는 첫 단계였다.

종류가 많은 브랜드·상품코드 다루기

High-cardinality

브랜드와 상품코드는 값의 종류가 너무 많다. 모든 값을 그대로 쓰면 모델이 우연한 이름을 외우기 쉬워서, 안정적으로 등장하는 값과 요약 통계 중심으로 바꿨다.

상위 빈도값과 구매 비율 만들기

Top-N · Count · Ratio

자주 등장하는 상위 N개 브랜드/카테고리를 고르고, 고객이 그 값에 몇 번 노출됐는지와 전체 거래 중 비율을 만들었다. “샀다/안 샀다”보다 “전체 구매 중 얼마나 차지하나”가 더 안정적이었다.

같이 산 상품 조합 보기

Co-purchase

상품 하나만 보면 애매한 경우가 많았다. 그래서 한 고객 안에서 어떤 상품군이 함께 등장하는지, 남성향/여성향/가족 구매가 섞이는지를 함께 봤다.

구매 흐름과 맥락을 숫자로 바꾸기

Sequence · Embedding

구매 로그를 문장처럼 보고 상품/브랜드를 단어처럼 보았다. 자주 같이 등장하는 상품은 가까운 숫자 좌표로 표현해서, 모델이 상품명을 몰라도 구매 맥락을 쓸 수 있게 했다.

비슷한 구매 패턴의 고객 참고하기

Graph · Neighbor

고객과 상품을 연결망으로 보고, 비슷한 상품을 산 고객들의 패턴을 보조 신호로 사용했다. 한 고객의 정보가 부족할 때 주변 구매 패턴이 약한 힌트가 됐다.

임베딩을 쉽게 말하면 — 상품마다 숫자 좌표를 찍는 것이다. 비슷한 고객 구매에서 자주 함께 등장하는 상품들은 가까운 위치에 놓이고, 관련이 적은 상품들은 멀어진다. 그래서 모델은 상품명을 직접 이해하지 않아도 구매 맥락을 숫자로 활용할 수 있다.
Top-N

브랜드/상품코드가 너무 많기 때문에, train에서 자주 등장하는 값만 골라 안정성을 확보했다.

Count·Ratio

특정 카테고리를 몇 번 샀는지와 전체 거래 중 비중을 같이 만들었다. 거래량이 많은 고객과 적은 고객을 비교하려면 비율이 중요했다.

OOF Evidence

성별 신호를 쓰는 feature는 validation fold의 정답을 보지 않도록 OOF 방식으로 만들었다. 누수를 막으면서 category의 성별 경향만 요약했다.

Selection

많이 만드는 것만으로는 부족했다. importance, drift, OOF 안정성을 보고 top200/top300/top800처럼 필요한 feature만 남겼다.

05Reference-Inspired

참고자료의 아이디어가
최종 순위 보정으로 이어졌다

제공된 참고자료의 핵심은 “임베딩 feature를 추가해보기”와 “중요 feature 주변은 보강하고, 하위 feature는 과감히 제거하기”였다. 최종 solution에서도 이 방향은 feature를 더 많이 쌓는 것보다 더 날카롭게 고르는 전략으로 이어졌다.

임베딩 feature 추가

W2V · FastText

구매 sequence를 문장처럼 보고 브랜드/상품을 단어처럼 다뤘다. W2V와 FastText는 단독으로 큰 모델을 이기지는 않았지만, 기존 source와 다른 순위 정보를 만들어 final blend에 보탬이 됐다.

  • 참고자료 실험 — W2V feature 추가 +0.00033
  • 참고자료 실험 — FastText feature 추가 +0.00089
  • baseline_top300 OOF 0.715833 → baseline_plus_fasttext 0.716034
  • 최종 유효 비중 — FastText 약 5.17%

feature를 조이고 다시 고르기

Tightening · Selection

참고자료에서 가장 크게 오른 방향은 중요 feature와 유사한 feature는 보강하고, 중요도 낮은 feature와 비슷한 feature는 제거하는 방식이었다. 우리도 top200/top300/top800, stable importance, group quota, R3 selected-feature branch로 이 방향을 반복했다.

  • 참고자료 실험 — 중요 feature 보강 + 하위 제거 +0.00202
  • 박중현 top800/stable top300 — 큰 feature pack에서 안정 feature만 선별
  • R3 — top200 LGBM/XGB를 다시 만들어 current를 보정
  • 최종 유효 비중 — R3 branch가 final의 70% 축을 형성

발표 포인트 — 참고자료 기반 feature는 “단독 1등 모델”이라기보다, 기존 강한 source가 놓친 순위를 보완해 최종 점수를 밀어 올린 역할을 했다.

결국 “새 feature를 무작정 늘리기”보다 “강한 feature를 남기고 약한 feature를 줄이는 것”이 private 1위에 더 직접적으로 연결됐다.

06Validation

OOF는 source를 믿어도 되는지 보는
내부 모의고사였다

여러 모델/source를 섞으려면 public 점수만 보는 것은 위험했다. 그래서 train 안에서 각 source가 실제로 도움이 되는지 OOF로 검증했다.

안 본 고객을 예측해보는 내부 검증

OOF Validation

train 고객을 5개 fold로 나누고, 4개 fold로 학습한 뒤 남은 1개 fold를 예측한다. 이 과정을 반복하면 모든 train 고객에 대해 “자기 자신은 학습에 쓰이지 않았을 때의 예측값”이 생긴다.

  • 답안지를 보고 맞힌 점수가 아니라 모의고사 점수에 가깝다
  • source를 섞기 전에 내부적으로 도움이 되는지 확인할 수 있다
  • 우리 문제는 AUC라서 예측 순위가 특히 중요했다

모델들의 순위표를 섞기

Rank Blend

모델마다 확률 스케일이 다르다. 어떤 모델은 전체적으로 0.3 근처를 많이 주고, 어떤 모델은 0.5 근처를 많이 줄 수 있다. 그래서 확률 자체보다 고객의 남성 가능성 순위를 맞춰 섞었다.

  • AUC는 절대 확률보다 순위에 민감하다
  • rank01로 각 source의 예측을 0~1 순위표로 바꿨다
  • 서로 다른 source를 더 안전하게 조합할 수 있었다
07
07Private Defense

강해 보여도 불안한 것은
그대로 믿지 않았다

최종 private 1위는 운 좋게 public만 맞춘 결과라기보다, 여러 source를 만들 때 안정성을 같이 확인한 결과였다. 특히 feature selection, drift 제한, OOF 검증, rank blend가 private 방어에 직접적으로 작동했다.

01

고객 단위 OOF로만 판단

Customer-level OOF

거래 한 줄 단위로 train/valid를 섞으면 같은 고객 정보가 새어 들어갈 수 있다. 그래서 모든 내부 검증은 고객 단위 fold로 보고, source를 섞기 전에도 OOF 기준으로 먼저 확인했다.

  • 같은 고객이 train/valid에 동시에 들어가지 않게 함
  • public 점수만 보고 비율을 계속 맞추는 위험을 줄임
  • public 최고였던 v12 32.5% 후보보다 v12 30% 후보가 private에서 근소하게 더 안정적이었다
02

feature를 많이 만들고 다시 줄임

Selection · Pruning

상위권 싸움에서는 feature가 많다고 무조건 좋지 않았다. top200/top300/top800처럼 feature 수를 바꿔 보고, 중요도와 안정성이 낮은 feature는 제거했다.

  • stable importance — 여러 모델에서 공통으로 중요한 feature 우대
  • group quota — 한 종류의 feature가 과하게 쏠리지 않게 제한
  • R3 — top200 LGBM/XGB로 current를 다시 보정
03

train/test 분포 차이를 경계

Drift-aware Selection

train에서는 좋아 보이지만 test 분포가 다르면 private에서 무너질 수 있다. 특히 박진성의 v9 branch는 featurebank 후보를 볼 때 성별 신호뿐 아니라 train/test drift도 같이 제한했다.

  • 성별 신호가 약하면 제외
  • train/test 분포 차이가 너무 크면 제외
  • public 최적 rank와 균등 rank를 섞어 private-safe source 생성
04

확률 평균 대신 순위 blend

Rank-map Blend

모델마다 확률 스케일은 다르지만 AUC는 순위가 중요하다. 그래서 최종 제출은 확률을 그대로 평균내지 않고 각 source를 rank01로 변환한 뒤 섞었다.

  • 스케일이 다른 모델을 공정하게 섞음
  • 특정 모델의 과한 확신값에 덜 끌려감
  • R3와 v12의 보완 순위를 안정적으로 결합

결과적으로 — public 최고점 하나만 따라간 것이 아니라, OOF·drift·feature 수·rank 안정성을 같이 봤기 때문에 private에서 하락폭을 줄이고 최종 1위를 지킬 수 있었다.

각자 만든 source를 공유하고

최종 solution은 한 사람이 만든 단일 모델이 아니라, 팀원들이 서로 다른 관점에서 만든 feature/source를 공유하고 OOF와 제출 결과를 기준으로 조합한 팀 단위 결과였다.

박중현
top800 / stable top300

큰 feature pack에서 안정적인 feature만 남기는 top800/stable top300 계열 source를 만들었다. 보수/공격 두 모델의 importance를 함께 보는 stable selection이 핵심이었고, 최종 tree에서는 team_refined로 들어가 R3 이전 current source의 중요한 축이 됐다.

박진성
feature_many / v9 rank

feature_many와 v9 private-safe rank source를 만들었다. featurebank 확장, drift-aware selection, 여러 seed와 균등 rank를 섞는 rank 안정화가 포함됐고, 최종 tree에서는 v9_opt_rank로 쓰였다.

유지원
v12 source

v12 late-stage model branch를 구성했다. 서준영이 만든 v61 parquet을 입력 feature 축으로 활용하고, W2V/FastText, RFM, Social/Dwell feature와 7-model Dirichlet blend를 결합해 최종 blend의 30% 보완 축을 만들었다.

서준영
feature source / integration

EDA 정리, v61 high-cardinality parquet, PhaseAD/A10A11, FastText/reference branch, R3 selected-feature branch를 만들고 — OOF 정렬, rank blend 후보 생성, 제출 검증까지 실험 전체를 연결했다.

협업 방식 — 각자 feature/source를 따로 만들고 끝낸 것이 아니라, 서로 결과물을 공유한 뒤 장점이 있는 부분만 다시 가져와 조합했다. 어떤 source는 단독 점수가 좋았고, 어떤 source는 단독보다 blend에서 강했다. 그래서 OOF, 상관관계, public/private 흐름을 함께 보며 팀 단위로 최종 조합을 만들었다.

실험 운영 — 여러 source를 한 폴더에 모아 custid 정렬, OOF alignment, rank 변환, 비율 탐색, submission 포맷 체크를 반복하는 운영 루프를 유지했다.

이 과정에서 단순히 점수가 높은 파일만 고른 것이 아니라, 팀원 source들이 서로 어디서 보완되는지 확인하고 최종 제출 후보를 좁혀 갔다.

def rank01(x):
    # AUC는 확률의 절댓값보다 순위에 민감함
    # 그래서 각 source를 0~1 순위표로 변환한 뒤 같은 스케일에서 blend
    return rankdata(x, method='average') / (len(x) + 1)

결론 — “좋은 모델 하나”가 아니라 “검증된 source의 조합”

이 solution의 핵심은 좋은 feature/source를 여러 개 만들고, 서로 보완되는 source만 검증해서 섞는 과정이었다. EDA로 방향을 정하고, 네 사람이 만든 source를 공유한 뒤, OOF와 rank blend를 이용해 최종 private 1위 submission을 만들었다.
